## Phase 1 — Plan (20 min)

**Why this matters:** This project brings together everything from Sessions 07–09: DOM manipulation, event handling, async/await, and API calls. It's also the first time you're making two sequential API calls — one depends on the result of the other.

**APIs you'll use — both free, no API key required:**
- Geocoding: `https://geocoding-api.open-meteo.com/v1/search?name=CITY_NAME&count=1`
- Weather: `https://api.open-meteo.com/v1/forecast?latitude=LAT&longitude=LON&current_weather=true`

**Plan the flow before coding:**
1. User types city name and clicks Search
2. Call geocoding API to get lat/lon
3. Call weather API with lat/lon
4. Display: city name, temperature (°C), wind speed, weather code description
5. Handle: city not found, network error

## Phase 2 — Build (85 min)

- **HTML**: A form with a text input and search button, a results `<div>` (hidden initially), an error `<div>` (hidden initially), a loading indicator
- **JavaScript**:
  1. Prevent form default submission
  2. Show loading state
  3. Fetch geocoding API — check that results exist
  4. Extract latitude and longitude from the first result
  5. Fetch weather API with those coordinates
  6. Hide loading, show results: city, temperature, wind speed
  7. In `catch`: hide loading, show error message
- **CSS**: Style the results card — it should look like a real weather widget
- **Bonus**: Map the `weathercode` number to a text description. Open-Meteo's docs list all codes.

## Review

What happens in your app if the user enters a city that doesn't exist? What if the network is offline? Test both scenarios in DevTools (Network tab → throttle to "Offline") and verify your error handling works.
